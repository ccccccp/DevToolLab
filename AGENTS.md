# DevToolLab 项目说明

## 语言
- 与用户交流时默认使用中文（简体）。
- 执行命令时不需要逐字翻译终端输出，但最终总结必须用中文。
- 只有在用户明确说“用英文”时才切换为英文。

## 当前状态
- 当前仓库处于初始化阶段，还没有正式代码结构。
- 本仓库现在按“单项目优先”推进，不再拆分独立的 `ui-components` 仓库。
- 所有组件先在主应用内沉淀，等复用边界稳定后，再决定是否抽成独立包。

## 项目目标
这是一个“内容站 + 工具站 + 轻 CMS + AI 内容处理流水线”的混合项目，目标是：

1. 做前台内容站，承接 AI 热点、专题、标签和文章详情页。
2. 做前台工具区，承接工具目录、工具详情、对比页和部分轻工具能力。
3. 做后台运营系统，支持文章、工具、采集源、任务和 SEO 管理。
4. 做内容生产流水线，支持抓取、去重、AI 摘要、人工审核和发布。

核心原则：
- 资讯负责拉新。
- 工具负责承接和变现。
- 后台负责提效。
- AI 负责辅助生产，不直接替代人工审核。

## 当前架构决策
在没有稳定代码和稳定复用需求之前，先收口为一套最小可执行方案：

- 前端：`Next.js App Router`
- 样式：`Tailwind CSS`
- 组件：优先使用应用内组件，不单独拆 UI 组件库
- 后端接口：当前使用 `Next.js Route Handlers`，后续需要边缘化部署时再迁移到 `Cloudflare Workers`
- 数据存储：目标为 `D1 + R2 + KV`
- 定时任务：目标为 `Cron Triggers`
- AI 能力：外部模型 API，先满足摘要和基础内容加工

当前执行原则：
- 先把站点跑起来，再考虑工程拆分。
- 先做同步、简单、可验证的实现，再考虑队列和异步流水线。
- 先做单仓内共享组件，再考虑独立包和版本管理。

## 目录规划
当前建议使用单仓结构，后续按此落地：

```txt
/
  app/                    # Next.js App Router 页面与路由
  components/
    ui/                   # 基础通用组件
    business/             # 业务组件
  lib/                    # 工具函数、配置、服务封装
  styles/                 # 全局样式和主题变量
  content/                # 初期可选：静态内容或示例数据
  docs/                   # 产品方案、架构设计、数据设计
  public/                 # 静态资源
```

约定：
- `components/ui` 放通用组件。
- `components/business` 放文章卡片、工具卡片、榜单模块等业务组件。
- 不要提前创建独立的 `packages/ui` 或 `DevToolLab-ui-components`。
- 如果未来确实要拆组件库，前提是至少有两个独立应用稳定复用同一批组件。

## MVP 范围
第一阶段只做最小闭环，不追求一步到位：

1. 首页
2. 文章列表页
3. 文章详情页
4. 工具列表页
5. 工具详情页
6. 最小后台录入能力
7. 2 到 3 个内容来源的抓取
8. AI 摘要
9. 人工审核后发布

明确不在第一阶段优先做的内容：
- 独立 UI 组件库
- 完整权限系统
- 复杂工作流编排
- 大而全的后台模块
- 过早的程序化 SEO 铺量
- 过度抽象的数据模型

## 页面优先级
前台优先级：
- `/`
- `/news`
- `/news/[slug]`
- `/tools`
- `/tools/[slug]`

后台优先级：
- `/admin/posts`
- `/admin/tools`
- `/admin/sources`
- `/admin/tasks`

这些页面先追求可用，再补视觉和复杂交互。

## 数据设计原则
数据表可以后续细化，但当前必须保证以下实体可落地：

- `posts`
- `tools`
- `tags`
- `sources`
- `crawl_items`
- `pipeline_jobs`
- `media_assets`

每个核心实体至少要有这些基础字段：
- `id`
- `slug`
- `status`
- `created_at`
- `updated_at`

涉及发布的内容还应优先考虑：
- `published_at`
- `source_url`
- `canonical_url`
- `summary`
- `review_status`

涉及任务的记录还应优先考虑：
- `job_type`
- `job_status`
- `retry_count`
- `error_message`

## 内容流水线原则
内容流水线先做简单闭环：

1. 抓取原文
2. 保存原始记录
3. 去重
4. AI 摘要
5. 人工审核
6. 发布

不要在初期同时做：
- 多级队列
- 自动改写全链路
- 全量自动打标
- 全自动发布

任何 AI 生成内容，默认都需要人工审核后再进入正式发布。

## 安全与风险
这个项目不只是普通博客，包含抓取和 AI 内容处理，所以必须注意：

- 后台登录必须使用安全 Cookie 或等价安全方案。
- 上传接口必须做校验。
- 高危操作必须有日志。
- 采集内容必须保留来源信息。
- 发布前必须允许人工审核。
- 不要默认把采集内容当作可直接商用内容。

## 文档规则
- `AGENTS.md` 只写当前执行约束和明确决策，不写大段愿景型方案。
- 产品规划、信息架构、数据库草案、SEO 方案，统一放到 `docs/`。
- 如果仓库结构或技术决策变了，优先更新本文件。

## 协作规则
- 修改前先确认当前仓库结构，不要假设历史目录仍然存在。
- 优先做最小可运行实现，不做过早抽象。
- 新增组件时，优先判断它是否真的会跨页面复用。
- 如果某个方案只是“未来可能需要”，默认先不做。
